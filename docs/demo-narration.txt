Welcome to the Model Observability Incident Platform. This project is a reliability control plane for model telemetry, incident state, release safety, and notification delivery. The dashboard is not a collection of decorative charts. The panel at the top is wired to the running FastAPI service, and every action changes durable state.

I will start with a degraded serving window. The population-shift, latency, and error controls generate a bounded forty-row reference and current dataset. When I run the evaluation, six policies execute. Feature drift, prediction drift, latency, and error rate fail, so the release decision changes to freeze release. Four open incidents appear with explicit severity and reason codes. Repeating this evaluation updates the same stable fingerprints instead of creating an incident storm. That distinction matters in a real on-call system: observed values are evidence, but they are not incident identity.

Under the interface, the evaluation is one SQLite write-ahead-log transaction. It commits the canonical request hash, evaluation result, incident mutation, immutable lifecycle event, and CloudEvents outbox row together. This removes the database-and-webhook dual-write gap. The delivery worker then claims due rows through expiring leases, preserves ordering for each incident, records every attempt, and writes to an idempotent receipt sink. Delivery is honestly described as at least once. Duplicate event IDs are accepted only when their payload digest matches; exhausted retries move to a dead-letter state.

The operator can acknowledge an incident with an expected version and transition key. Stale versions and conflicting replays are rejected. Recovery is also deliberate. One healthy window records evidence, but does not resolve the alert. Two consecutive healthy windows satisfy the hysteresis policy, auto-resolve the incidents, emit ordered recovery events, and return the release decision to continue. The worker drains those new events until pending and in-flight counts are both zero.

This evidence view exposes the engineering guarantees behind the workflow. Evaluation and lifecycle replay pass after restart. Trace propagation preserves the incoming W3C trace identifier. Prometheus metrics use bounded labels, while request IDs, incident IDs, model versions, and raw features are deliberately excluded from metric dimensions. The notification contract separately proves lease takeover, stale-worker fencing, per-incident ordering, immutable attempt history, and the dead-letter path.

The repository also includes structured JSON logs, OpenTelemetry spans, Prometheus metrics, an Airflow three-point-three DAG parsed against the real public SDK, and Kubernetes architecture labs for disruption controls, workload identity, network policy, resource envelopes, autoscaling, queueing, and release admission. The README labels each capability as executable, integrated, or design-only, so a manifest is never presented as a running cluster.

The responsive operator view keeps the same controls and state on a narrow screen. For a portfolio review, the important signal is the boundary of the claim. SQLite makes transaction and crash behavior inspectable locally. The production migration is explicit: Postgres for high availability, an external broker and relay, authenticated incident transitions, tenant policy ownership, calibrated thresholds, retention, and restore drills. Tests cover idempotency, schema migration, drift, freshness, concurrency, retries, recovery, cardinality, and real HTTP behavior. This is a compact local system, but its failure semantics are designed to stand up to senior-level review.
